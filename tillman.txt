  M1 → M2 is the critical dependency. You can't test
  Blend math without real ledger data flowing. M3 is

  lets you validate the pipeline works before layering
   on Blend's complexity.

  de-risk, you could split it into M2a (processor
  interface + contract events + basic Blend positions)
   and M2b (health factor parity + oracle + backfill +
   TimescaleDB snapshots). That way you get Blend
  positions visible in Postgres before perfecting the
  math.

  Repository Structure

  I'd recommend 3 repositories for the full delivery:

  Repo: orion
  What: Monorepo — cmd/orion-ingest, cmd/orion-api, shared internal/ (models, adapters, db, pipeline)
  Why separate: The two binaries share >70% of their code: models, database layer, adapter interfaces, config
    parsing. Splitting them would mean duplicating types or creating a third shared-lib repo. Go monorepos with
    multiple cmd/ entrypoints are idiomatic.
  ────────────────────────────────────────
  Repo: orion-sdks
  What: Auto-generated TypeScript, Go, Python SDKs from OpenAPI spec
  Why separate: Generated code has a different release cadence and CI pipeline (openapi-generator, npm publish,
    pypi publish, go module tagging). Mixing generated code with handwritten Go creates noise in diffs and
    reviews.
  ────────────────────────────────────────
  Repo: orion-docs
  What: Developer portal, API docs, integration guides
  Why separate: Keeps documentation deploy (likely static site) independent from binary releases. Optional —
  could
    start as a /docs folder in the monorepo and split later if it grows.

  Adapter Repo Strategy                        
                                                                                                                 
  Don't split adapters into their own repo. Here's why:                                                          
                                                                                                                 
  In Go, adapters share the processor interface, Stellar SDK helpers, XDR decoding, the pipeline's Message type, 
  and test infrastructure. Separate repos means either:                                                          
  - A orion-adapter-sdk shared library repo (now you have 3+ repos to version-lock), or                          
  - Duplicated types and import cycle pain                                                                       
                                          
  The pattern that works (Terraform providers, Grafana datasources, Prometheus exporters) is: adapters live in   
  the ingest repo, community adapters can be external repos importing the interface package.                     
                                                                                                                 
  Here's what I'd recommend:                                                                                     
                                               
  ┌──────────────┬────────────┬───────────────────────────────────────────────────────────────────────────────┐  
  │     Repo     │ Visibility │                                   Contains                                    │
  ├──────────────┼────────────┼───────────────────────────────────────────────────────────────────────────────┤  
  │ orion-ingest │ Public     │ Pipeline engine + adapter interface + Blend/Aquarius/Soroswap adapters. The   │
  │              │            │ "indexer/mirror/backfill" tool.                                               │
  ├──────────────┼────────────┼───────────────────────────────────────────────────────────────────────────────┤  
  │ orion-api    │ Private    │ REST server, rate limiting, API keys, response envelope. Imports models from  │
  │              │            │ orion-ingest.                                                                 │  
  ├──────────────┼────────────┼───────────────────────────────────────────────────────────────────────────────┤
  │ orion-sdks   │ Public     │ Auto-generated TS/Go/Python SDKs from OpenAPI spec                            │  
  ├──────────────┼────────────┼───────────────────────────────────────────────────────────────────────────────┤
  │ orion-web    │ Private    │ Frontend / developer portal                                                   │
  ├──────────────┼────────────┼───────────────────────────────────────────────────────────────────────────────┤  
  │ scaffold     │ Private    │ What we're in now — validation workspace, reference lift                      │
